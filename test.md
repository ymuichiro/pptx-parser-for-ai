# テスト基準（Test Standard）

## 1. 目的
- 本ドキュメントは `pptx-parser-for-ai` の品質を継続的に保証するためのテスト基準を定義する。
- 対象は機能正当性に加え、セキュリティ、パフォーマンス、堅牢性、再現性を含む。
- 本基準を満たさない変更はマージしない。

## 2. 適用範囲
- 対象モジュール:
  - `types`
  - `parser`
  - `theme`
  - `layout`
  - `renderers`
  - `qa`
  - `index`（公開 API）
- 対象外:
  - AI 推論・モデル連携機能（本リポジトリでは実装しない）

## 3. テストレベル定義

## 3.1 Unit Test（最小単位）
- 目的: 純粋ロジックと境界条件を高速に検証する。
- 実行要件: 全 PR で必須。
- 対象:
  - スキーマ検証
  - 正規化ロジック
  - レイアウト計算
  - 各レンダラのオプション生成
  - エラー型変換

## 3.2 Integration Test（モジュール連結）
- 目的: DSL 入力から PPTX 生成までの経路を検証する。
- 実行要件: 全 PR で必須。
- 対象:
  - `generateFromFile`
  - `generate`
  - テーマ解決
  - 複数スライド構成

## 3.3 Regression Test（再発防止）
- 目的: 過去バグの再発を防止する。
- 実行要件: バグ修正 PR では必須。
- 対象:
  - 再現 DSL fixture
  - 最小再現ケース
  - 期待エラーコードとメッセージ

## 3.4 Security Test（攻撃耐性）
- 目的: 境界入力と悪性入力への耐性を検証する。
- 実行要件: 日次 CI + リリース前必須。
- 対象:
  - パストラバーサル
  - 異常 YAML
  - 巨大入力
  - 不正 URL
  - 依存関係脆弱性

## 3.5 Performance Test（性能回帰）
- 目的: 速度・メモリの回帰を検知する。
- 実行要件: main マージ時 + リリース前必須。
- 対象:
  - parse/validate/normalize
  - layout
  - render
  - end-to-end 生成時間

## 4. 品質ゲート（数値基準）

## 4.1 カバレッジ
- 全体ステートメントカバレッジ: 85%以上。
- 全体ブランチカバレッジ: 80%以上。
- クリティカル経路（`parser`, `layout`, `renderers`, `index`）:
  - ステートメント: 90%以上
  - ブランチ: 85%以上

## 4.2 成功条件
- `lint`、`typecheck`、`unit`、`integration`、`security`、`build` が全て成功。
- High/Critical の既知脆弱性が 0 件。
- パフォーマンス回帰がベースライン比 10% 未満。

## 4.3 失敗条件
- 1 件でも flaky 判定が出た場合は原因調査を優先し、恒久対策前の再実行マージは禁止。
- TODO コメント付きでの暫定緩和は不可（明示承認がある場合のみ例外）。

## 5. モジュール別必須テスト

## 5.1 parser
- 正常系:
  - 最小 DSL が正しく parse/validate/normalize される。
  - デフォルト値補完が仕様どおりに行われる。
- 異常系:
  - 必須項目欠落
  - 型不一致
  - 未知フィールド
  - 上限超過（文字列長・配列長・ネスト深度）
  - 不正 YAML 構文
- 期待:
  - `ParseError` または `ValidationError` が型付きで返る。

## 5.2 theme
- 正常系:
  - 組み込みテーマ読込
  - カスタムテーマ読込
  - 色・タイポ・レイアウトの参照解決
- 異常系:
  - 不正テーマパス
  - 不足トークン
  - 不正色コード
- 期待:
  - `ThemeError` で原因を保持。

## 5.3 layout
- 正常系:
  - `auto`, `single-column`, `two-column`, `three-column`
  - 図要素優先分配ロジック
- 異常系:
  - 要素 0 件
  - 極端に多い要素
  - 不正 bounds
- 性質テスト:
  - すべての要素がスライド境界内
  - 要素間重なり率が許容範囲内
  - 同一入力で同一結果（決定性）

## 5.4 renderers
- 正常系:
  - `text`, `bullet-list`, `table`, `chart`, `image`, `network-diagram`, `flowchart`
  - 必須スタイル適用
- 異常系:
  - 欠損データ
  - 参照不能リソース
  - 長文・多要素
- 期待:
  - 失敗時に `RenderError` を返すか、仕様に定めたフォールバック動作を行う。

## 5.5 qa
- 正常系:
  - 問題検出
  - 自動修正
  - 修正後再検証
- 異常系:
  - 修正不能ケース
  - 最大反復回数到達
- 期待:
  - 無限ループしない
  - エラー分類が保持される

## 5.6 公開 API（index）
- `generateFromFile` と `generate` の契約を検証。
- 成功時の `GenerationResult` 構造を検証。
- 例外時に入力情報の機密が漏洩しないことを検証。

## 6. 出力検証（PPTX）
- binary 全体の完全一致比較は禁止（ZIP メタデータ差分のため不安定）。
- 構造検証を行う:
  - スライド数
  - 各スライドの主要テキスト
  - 図形・画像・表の存在数
  - 主要座標が期待範囲内
- 可能なら XML 正規化後にスナップショット比較を行う。

## 7. テストデータ基準
- `tests/fixtures` に用途別 DSL を配置:
  - `valid/`
  - `invalid/`
  - `security/`
  - `performance/`
- fixture は最小再現を優先し、1 fixture 1 意図で作成する。
- 外部ネットワーク依存 fixture は禁止（ローカル再現可能にする）。

## 8. セキュリティテスト基準
- 必須ケース:
  - `../../` を含むパス入力拒否
  - symlink 迂回拒否
  - YAML alias/anchor による過剰展開耐性
  - 異常に長い文字列・巨大配列入力時の安全停止
  - URL 読込が無効設定で拒否されること
- 依存検査:
  - `npm audit --omit=dev` 相当
  - SBOM 生成可能な構成を維持

## 9. パフォーマンステスト基準
- ベンチマーク入力を固定し、計測値は 5 回以上の中央値を採用。
- 測定指標:
  - parse/validate/normalize 時間
  - layout 時間
  - render 時間
  - end-to-end 時間
  - ピークメモリ
- 判定:
  - 直近 main ベースライン比 10% 以上の劣化は失敗。
  - メモリ上限超過は失敗。

## 10. 信頼性テスト基準
- Fuzz テストを導入し、ランダム DSL でパーサ/バリデータを検証する。
- 目安:
  - PR 時: 300 ケース以上
  - 夜間: 3,000 ケース以上
- 期待:
  - クラッシュしない
  - 未分類例外を出さない

## 11. テスト実行ポリシー（CI）
- PR 必須ジョブ:
  - `lint`
  - `typecheck`
  - `test:unit`
  - `test:integration`
  - `test:coverage`
- main/Release 必須追加ジョブ:
  - `test:security`
  - `test:perf`
  - `test:fuzz`（夜間でも可）

## 12. Flaky Test 対応
- flaky を検知したテストは `@flaky` タグで隔離し、24 時間以内に恒久対策 PR を作成。
- flaky の再実行による通過を成功扱いしない。
- 原因が環境依存の場合はテストを deterministic に修正する。

## 13. 回帰管理ルール
- すべての不具合チケットに再現テスト ID を紐付ける。
- 修正 PR は「再現テスト -> 修正 -> 成功テスト」の順でコミットする。
- 回帰テストが追加されないバグ修正 PR は原則却下。

## 14. 完了の定義（Definition of Done）
- 機能追加時:
  - 正常系 Unit/Integration テストが追加済み
  - 異常系テストが追加済み
  - 既存テストを壊していない
- 不具合修正時:
  - 再現テストが先に追加済み
  - 修正後に再発しないことを確認済み
- リリース時:
  - 品質ゲートを全通過
  - テストレポートを保存
