# 開発ガイドライン（Security / Performance / Robustness）

## 1. 目的
- 本ガイドラインは、`pptx-parser-for-ai` の実装に脆弱性・性能劣化・不安定動作が入り込まないための必須基準を定義する。
- 本リポジトリの責務は「抽象化された DSL から PPTX を生成するライブラリ」であり、AI 機能は対象外とする。

## 2. 基本原則
- Secure by Default: 安全設定をデフォルトにし、危険機能は明示的に opt-in。
- Fail Closed: 不正入力・不明値は黙って通さず、明示的エラーで停止。
- Deterministic: 同一入力 + 同一バージョンでは同一結果を返す。
- Small Surface: 公開 API と I/O 境界を最小化し、内部実装は純粋関数を優先。
- Measurable Quality: すべての品質要件を数値で評価可能にする。

## 3. 適用範囲と責務境界
- `parser`: DSL の構文解析・検証・正規化のみ。
- `theme`: テーマ読み込みと参照解決のみ。
- `layout`: 座標計算のみ（副作用禁止）。
- `renderers`: `pptxgenjs` への描画命令生成のみ。
- `qa`: 出力検証と修正提案のみ。
- I/O は `generateFromFile`・テーマ読込・最終出力以外で禁止。

## 4. セキュリティ基準（必須）

### 4.1 入力検証
- DSL/Theme はスキーマで厳格検証し、未知フィールドはエラーとする（将来拡張時は明示的な互換モードを追加）。
- 必須フィールド欠落・型不一致・範囲外の値は `ValidationError` を返す。
- 文字列長、配列長、ネスト深度に上限を設ける。
  - 例: 文字列 10,000 文字、配列 10,000 要素、ネスト深度 20。

### 4.2 ファイルパス・外部入力
- 相対パスは `path.resolve` 後に許可ディレクトリ配下であることを検証し、パストラバーサル（`..`）を拒否。
- シンボリックリンク解決後も許可ディレクトリ外への脱出を拒否。
- URL 画像読込はデフォルト無効。将来有効化する場合は以下を必須化:
  - `https` のみ許可
  - タイムアウト、サイズ上限、リダイレクト回数上限
  - プライベート IP / メタデータエンドポイントへのアクセス拒否（SSRF 対策）

### 4.3 YAML/データパーサ
- 安全な YAML ローダ設定を使用し、任意コード実行につながるタグ/カスタム型を禁止。
- 受け取った値は `unknown` から型ガードで絞り込む。

### 4.4 出力安全性
- テキスト内容は XML/PPTX 構造破壊を起こさないようエスケープ処理を統一。
- 出力ファイルは一時ファイルに書き込み後、原子的 rename で確定。
- 失敗時に壊れた中間ファイルを残さない。

### 4.5 依存関係
- 依存追加時は用途・代替不可理由・メンテナンス状況を PR に明記。
- lockfile を必須化し、CI で差分検証。
- `npm audit` または同等ツールで High/Critical 脆弱性ゼロをリリース条件にする。
- 不要な実行時依存は追加しない（特に `react`, `react-dom` などは要件適合時のみ）。

## 5. パフォーマンス基準（必須）

### 5.1 複雑度要件
- `parse`/`validate`/`normalize` は入力サイズに対して原則 O(n)。
- レイアウト計算は要素数に対して線形または準線形を基本とする。
- `force-directed` のような反復系は最大反復回数と収束条件を必須化。

### 5.2 リソース上限
- 無制限ループ・無制限再帰・無制限再試行を禁止。
- 画像処理・チャート処理にサイズ上限を設ける。
- 1 回の生成処理で利用可能なメモリ目標値を定義し、超過時は明示エラーで停止。

### 5.3 計測と回帰検知
- CI に性能回帰テストを導入し、基準ケースで前バージョン比 10% 超の劣化を失敗とする。
- 計測対象:
  - パース時間
  - レイアウト時間
  - 描画時間
  - ピークメモリ
- ベンチマークは固定入力を使用し、ノイズを避けるため複数回中央値で判定。

## 6. 堅牢性基準（必須）

### 6.1 エラーモデル
- 想定エラーは型付きで分類する:
  - `ParseError`
  - `ValidationError`
  - `ThemeError`
  - `LayoutError`
  - `RenderError`
  - `IOError`
- `throw new Error("...")` の乱用を禁止し、上位で判定可能なエラー型を使う。

### 6.2 失敗時挙動
- `catch` で握りつぶさない。必ず原因を保持して再送出または変換。
- 部分失敗時の挙動を明示する（例: 該当要素のみスキップして警告、またはスライド全体失敗）。
- 自動修正（QA）は反復回数上限を持ち、無限再帰を禁止。

### 6.3 決定性と再現性
- 日時・乱数・環境依存情報をロジックへ直接埋め込まない。
- 必要な乱数は seed 指定可能にする。
- テスト環境で同一入力のスナップショット差分が安定すること。

## 7. 実装規約（TypeScript）
- `tsconfig` は `strict: true` を必須。
- `any` 禁止（やむを得ない場合は理由コメント + 局所化）。
- `switch` は `never` による網羅性チェックを必須化。
- 公開 API の引数/戻り値は明示型必須。
- ミュータブル共有状態を避け、可能な限り不変データで処理。
- 同期 I/O（`fs.readFileSync` など）は CLI 初期化以外で禁止。

## 8. テスト戦略と品質ゲート

### 8.1 テスト必須項目
- Unit Test:
  - パーサ、バリデータ、正規化、レイアウト、各レンダラ
- Integration Test:
  - DSL 入力から `.pptx` 生成まで
- Regression Test:
  - バグ修正には再発防止テストを必須追加
- Security Test:
  - パストラバーサル、異常 YAML、巨大入力、無効URL

### 8.2 最低ライン
- 主要モジュールのステートメントカバレッジ 85% 以上。
- クリティカル経路（parse/validate/layout/render）は分岐カバレッジ 80% 以上。
- CI で `lint` `typecheck` `test` `security scan` `build` 全通過を必須化。

## 9. ログと可観測性
- ログは構造化し、機密情報（ファイル実体、トークン、秘密鍵等）を出力しない。
- エラーには以下を含める:
  - エラー種別
  - 発生フェーズ
  - 入力位置（可能ならスライド/要素インデックス）
- デバッグログはデフォルト無効、明示フラグで有効化。

## 10. コードレビュー基準（PR チェックリスト）
- 入力検証が境界で行われているか。
- 上限値（サイズ/反復/深度）が定義されているか。
- エラー型が適切か、握りつぶしがないか。
- 複雑度が許容範囲か（最悪計算量を説明できるか）。
- テストが正常系/異常系/回帰をカバーしているか。
- 依存追加の妥当性と脆弱性状況が確認されているか。

## 11. 禁止事項
- `eval` / `new Function` / 動的コード実行
- 無制限 retry / 無制限再帰 / 無制限メモリ確保
- 非検証の外部入力をそのまま描画・パス解決・オプション反映
- 暗黙のフォールバックで異常を隠す実装
- テストなしのクリティカルロジック変更

## 12. リリース判定
- High/Critical の既知脆弱性 0 件
- 性能回帰閾値以内（10% 未満）
- 主要機能の統合テスト成功
- 破壊的変更は `CHANGELOG` と移行手順を記載

## 13. 運用ルール
- セキュリティ問題を発見した場合は公開 Issue ではなくメンテナに非公開連絡する。
- 重大不具合は最小修正で hotfix を作成し、その後に根本対策を別 PR で実施する。
- 本ガイドラインに抵触する実装は、動作していてもマージしない。
